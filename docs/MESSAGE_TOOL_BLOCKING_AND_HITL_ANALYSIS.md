# `message` 工具调用后是否阻塞 Agent，以及是否应该增加阻塞机制

本文档回答两个核心问题：

1. 当前 `nanobot` 中，主 Agent 调用 `message` 工具后，是否会阻塞整个 Agent 系统运行？
2. 如果当前不阻塞，用户后续补充信息还有什么价值？
3. 是否建议加入“Agent 提问后，在用户补充信息前阻塞执行”的机制？

本文讨论的“阻塞”主要指：

- Agent 在向用户发出追问或确认请求后
- 是否停止继续往后执行
- 是否进入显式等待状态
- 是否必须等用户回复后才能恢复

---

## 1. 结论先说

### 1.1 当前 `message` 工具调用后 **不会自动阻塞整个 Agent**

当前实现里，`message` 工具的行为只是：

- 发送一条 `OutboundMessage`
- 返回一个普通字符串结果，例如：
  `Message sent to feishu:chat123`

它本质上仍然只是一次普通工具调用。

也就是说：

- Agent 调用了 `message`
- 工具执行成功
- 工具结果被写回消息历史
- Agent Loop 正常继续下一轮推理

系统不会因为调用了 `message` 就自动进入“等待用户回复”的挂起态。

---

### 1.2 当前的用户补充信息仍然是有价值的

虽然 `message` 不会自动阻塞，但用户补充信息并不是没有价值。

它的价值主要来自两类机制：

1. **mid-turn injection**
   如果用户回复足够快，而且当前 session 正在运行，这条回复可能被中途注入到当前执行中

2. **后续 session 继续**
   如果当前这一轮已经结束，用户的新回复会作为同一 session 的下一轮输入，Agent 会基于已有会话历史继续

所以当前系统并不是：

- “问完以后完全不管用户”

而是：

- “问完以后默认继续跑，用户的新输入如果来得及时可以中途并入；如果来得稍晚，则在下一轮继续生效”

---

### 1.3 是否建议加阻塞机制？

**建议加，但不建议做成“所有 `message` 调用一律阻塞”的粗暴机制。**

更合理的建议是：

- 保持当前 `message` 工具默认不阻塞
- 新增显式的“请求输入 / 请求批准并挂起”机制
- 只对需要人工参与才能继续的场景启用阻塞

换句话说：

**应该增加“选择性阻塞”，不应该增加“无差别阻塞”。**

---

## 2. 当前 `message` 工具到底做了什么

先看 `message` 工具自身的实现。

在 `nanobot/agent/tools/message.py` 中，`MessageTool.execute(...)` 会：

1. 解析参数
2. 组装 `OutboundMessage`
3. 调用 `send_callback`
4. 成功后返回字符串：
   `Message sent to channel:chat`

关键点是：

- 它没有设置任何 “waiting” 状态
- 没有触发挂起
- 没有主动停止 Agent Loop
- 没有等待用户回包的逻辑

所以它只是：

**一个发送消息的工具，不是一个等待输入的同步原语。**

这和很多工作流系统里的 `ask_user()` / `await_approval()` 是两回事。

---

## 3. 从 Agent Runner 视角看，为什么它不会阻塞

`message` 在 Agent Runner 看来只是普通工具。

在 `nanobot/agent/runner.py` 中：

1. 模型先输出工具调用
2. Runner 执行工具
3. 把工具结果写入 `messages`
4. 继续下一轮 LLM 请求

这条逻辑对 `message` 和对 `read_file`、`grep`、`exec` 并没有本质区别。

也就是说在运行时层面：

- `message` 没有特殊语义
- 它不会触发挂起分支
- 也不会改变执行状态机

因此：

**“调用了 `message`” ≠ “系统开始等用户”。**

---

## 4. 那为什么之前文档会说 Agent 可以向用户追问

因为“能追问”和“会阻塞等待”不是一回事。

当前系统支持：

- Agent 主动给用户发追问消息
- 用户回复后继续同一 session
- 长任务中吸收 follow-up

但当前系统不支持：

- 调用 `message` 后自动停住并等待

所以你可以把当前机制理解成：

**异步追问**

而不是：

**同步等待**

---

## 5. 如果不阻塞，用户补充信息为什么仍然有意义

这是最关键的问题。

表面上看，好像逻辑是：

1. Agent 问用户一个问题
2. Agent 自己继续往下走
3. 那用户现在补充的内容还有什么用？

答案是：有用，但效果取决于回复到达的时机。

---

## 6. 情况一：用户回复得很快，当前任务还在跑

当前项目有一个非常重要的机制：

- per-session pending queue
- mid-turn injection

在 `AgentLoop` 中，如果同一 session 已经有任务在运行：

- 新消息不会立刻开新任务
- 优先被路由进 pending queue

之后 `AgentRunner` 会在多个检查点 drain 这些 follow-up 消息，并把它们作为新的 user message 注入当前 messages。

这意味着：

如果 Agent 已经发出追问，用户几秒内就回复了，而此时任务还没结束，那么：

- 这条用户补充信息很可能可以直接影响当前执行
- Agent 后续步骤会基于它继续推理

所以在这个场景下，用户补充信息的价值非常高。

---

## 7. 情况二：用户回复较晚，当前任务已经结束

如果当前这一轮已经结束，那么用户补充信息不会再改变那个已经结束的执行路径。

但它依然有意义，因为：

- 新输入会进入同一个 session
- Agent 会基于历史继续下一轮

这个时候价值表现为：

- 修正上一轮的不完整信息
- 触发下一轮更精确的执行
- 继续未完成流程

例如：

1. Agent 问：“请提供网格尺寸。”
2. 当前轮结束
3. 用户 10 分钟后回复：“2mm”
4. 下一轮 Agent 会基于已有会话上下文继续往下走

这个过程不是“原地恢复执行栈”，但仍然是有效的人机协作。

---

## 8. 情况三：Agent 问完后仍然继续做了一些不依赖用户输入的工作

这其实是当前架构下最合理的默认行为。

因为很多时候：

- Agent 发问，并不意味着后面的所有事情都必须停止

例如：

- 问用户要材料参数
- 但与此同时，Agent 还能继续扫描文件结构
- 还能检查现有数据
- 还能准备脚本模板
- 还能做前置分析

在这种情况下，不阻塞是更高效的。

所以“问完之后继续走”并不一定是缺点，关键在于：

**继续走的步骤是否真的依赖用户还没补充的信息。**

---

## 9. 当前不阻塞机制的优点

当前设计不是随便写的，它确实有一组明显优点。

### 9.1 吞吐量更高

Agent 不会因为发出一条问题就立即挂起。

它还能继续：

- 做不依赖用户输入的分析
- 完成外围准备工作
- 让整体任务更快收敛

### 9.2 更适合自然聊天式体验

不是所有用户消息都应该触发工作流式暂停。

对于很多普通场景：

- 先问一句
- 如果用户稍后补充，再下一轮继续

已经足够自然。

### 9.3 实现复杂度低

如果 `message` 自动阻塞，就需要：

- waiting 状态
- 恢复 token
- 用户回复路由
- 超时策略
- 并发控制

而当前实现避免了这套复杂系统。

---

## 10. 当前不阻塞机制的缺点

你的担心完全成立：不阻塞也确实会带来明显问题。

### 10.1 可能继续执行到“其实不该继续”的步骤

如果用户输入对后续步骤是硬依赖，而 Agent 仍然继续执行，就可能出现：

- 用默认值瞎猜
- 走错分支
- 执行无效操作
- 做出需要回滚的结果

这是当前机制最大的风险。

### 10.2 用户感觉“我还没回答，你怎么已经往下做了”

从用户体验角度，这会显得不一致。

尤其在下面这些场景里问题很大：

- 是否执行危险命令
- 是否覆盖文件
- 是否提交代码
- 是否运行仿真

这些场景更适合“明确等待确认”。

### 10.3 追问和执行之间缺少硬约束

当前系统只能靠：

- prompt 约束
- skill 约束
- 模型自身判断

来决定问完后是否继续。

这不是强工程约束。

---

## 11. 当前机制下，什么时候用户补充信息“没什么价值”

这确实会发生，主要在以下情况。

### 11.1 后续关键路径已经走完

如果 Agent 在问完之后已经做完所有关键决策，那么用户再补充：

- 只能影响下一轮
- 无法逆转刚才的执行路径

### 11.2 问题本身其实是“必须同步回答”的

例如：

- “是否允许执行这个命令？”
- “请从 A/B 两个方案里选一个。”
- “请输入必须参数 mesh_size。”

这类问题本质上是 gating question。

如果不阻塞，就会导致补充信息价值下降。

### 11.3 子 Agent 场景

subagent 没有 `message` 工具，不能自己问用户。

如果 subagent 发现缺东西，只能把问题带回主 Agent。

这进一步说明：

当前系统更适合“主 Agent 层面对人类做异步追问”，不适合“每个子任务自己精确等待人工”。

---

## 12. 是否建议增加阻塞机制

我的建议是：

**建议增加，但必须是显式、选择性的阻塞机制。**

绝对不建议把现有 `message` 改成“一调用就阻塞”。

原因如下。

---

## 13. 为什么不建议让 `message` 默认阻塞

如果你把 `message` 改成默认阻塞，会带来很多负面影响。

### 13.1 它会破坏 `message` 作为普通消息发送工具的语义

当前 `message` 不只是追问用户，也用于：

- 发送通知
- 发送附件
- 发送结果
- 跨 channel 发送内容

这些都不应该自动让 Agent 挂起。

所以不能把：

- “给用户发消息”

和

- “请求输入并阻塞等待”

混为一谈。

### 13.2 它会让很多正常场景变慢

有些消息只是：

- 提示
- 进度通知
- 结果发送

这类场景如果也触发阻塞，会让系统体验变差。

### 13.3 会产生大量歧义

例如 Agent 发一句：

- “我先开始分析，有问题再问你。”

这到底要不要阻塞？

如果工具层没有更细的语义，系统就无法判断。

---

## 14. 更合理的做法：引入新的“阻塞式人机交互原语”

更好的设计不是改造 `message`，而是新增明确语义的机制。

例如引入：

- `request_user_input(...)`
- `request_approval(...)`
- `await_user_reply(...)`

这些工具/动作的语义应该是：

1. 向用户发出结构化请求
2. 让当前执行进入等待状态
3. 直到用户回复后再恢复

这和 `message` 的区别应该非常明确：

- `message` = 发消息，不阻塞
- `request_user_input` = 发问题，并挂起等待
- `request_approval` = 发审批请求，并挂起等待

这才是工程上干净的设计。

---

## 15. 我建议增加的阻塞模型

如果你准备往下做，我建议采用下面这个模型。

### 15.1 运行态新增 waiting 状态

例如：

- `running`
- `awaiting_user_input`
- `awaiting_approval`
- `resuming`
- `completed`
- `failed`

### 15.2 等待请求必须结构化

例如：

```json
{
  "type": "user_input_request",
  "request_id": "req_123",
  "question": "请提供网格尺寸，例如 2mm。",
  "required_fields": ["mesh_size"],
  "resume_mode": "same_task"
}
```

### 15.3 用户回复必须能和等待点绑定

也就是说系统不能只知道“用户说了一句话”，还要知道：

- 这句话是在回答哪个等待请求

### 15.4 恢复不能靠自然会话猜测

应该有：

- `resume_token`
- `request_id`
- `waiting_context`

来恢复原执行流。

---

## 16. 是否建议在当前项目里增加这个机制

### 短期建议

建议先增加：

- 显式的 `request_user_input`
- 显式的 `request_approval`

但先只在 **主 Agent** 层实现，不要直接下放到 subagent。

原因是主 Agent 已经具备：

- 用户对话入口
- session
- pending queue
- checkpoint
- 恢复逻辑

所以主 Agent 是实现阻塞式 HITL 的最好落点。

### 中期建议

等主 Agent 层的等待-恢复机制稳定后，再考虑让 subagent 通过结构化返回请求主 Agent 代为发起等待。

也就是：

- subagent 发现需要人工
- subagent 返回 `needs_user_input`
- 主 Agent 调 `request_user_input`
- 用户回复后主 Agent 恢复子流程

### 不建议

不建议一步到位地：

- 让所有 `message` 调用自动阻塞
- 或者让 subagent 直接具备阻塞式 HITL

这两条都会把当前系统复杂度拉得很高。

---

## 17. 对你这个问题的直接回答

### 问题 1：当前 `message` 工具调用后会阻塞整个 Agent 系统吗？

不会。

它只是普通工具调用，发送消息后返回字符串结果，Agent Loop 会继续。

### 问题 2：如果不阻塞，用户补充信息还有什么价值？

有价值，主要体现在：

- 用户回复快时，可通过 mid-turn injection 并入当前执行
- 用户回复晚时，可作为同一 session 的下一轮输入继续任务
- Agent 在等待期间还可以先做不依赖该输入的工作

但你的担心也成立：

- 如果这个问题是强依赖 gating question，不阻塞就会显著降低补充信息的即时价值

### 问题 3：是否建议加上“Agent 提问后，用户补充前的阻塞机制”？

建议加，但应该：

- 作为新机制加
- 不是修改现有 `message` 的默认语义
- 只在明确需要等待用户输入/批准的场景启用

---

## 18. 一句话总结

当前 `message` 工具是：

**异步通知/追问工具**

不是：

**同步等待/阻塞恢复工具**

所以当前系统支持“问用户”，但不原生支持“问完后自动挂起等待”。

如果你要把 HITL 做扎实，正确方向不是把 `message` 变成阻塞工具，而是新增显式的：

- `request_user_input`
- `request_approval`
- `resume`

这类结构化等待-恢复机制。
