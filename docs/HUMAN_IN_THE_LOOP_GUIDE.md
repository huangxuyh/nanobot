# nanobot 项目的 Human-in-the-Loop 机制详解

本文档基于当前仓库实现，系统分析 `nanobot` 是否支持 Human-in-the-Loop（HITL），以及它具体支持到什么程度。

这里讨论的 HITL 主要指两类场景：

1. Agent 在执行过程中发现信息不足，需要用户补充输入
2. Agent 在执行关键操作前，需要用户确认、授权或批准后才能继续

本文不讨论外部平台的账号 OAuth、机器人权限申请等部署层问题，而聚焦于 Agent 运行时的人机协作机制。

---

## 1. 结论先说

当前项目 **支持“会话式 Human-in-the-Loop”**，但 **不支持“强约束、结构化、系统级审批流式 Human-in-the-Loop”**。

更具体地说：

- 支持 Agent 主动向用户发消息追问
- 支持用户补充消息后继续当前会话
- 支持在 Agent 还没完全结束这一轮时，把用户的新消息中途注入到当前运行中
- 支持中断恢复，避免等待用户或中途异常时丢失上下文

但是：

- 没有显式的 `awaiting_user_input` 状态机
- 没有系统级 “此工具调用必须先经过用户批准” 的拦截器
- 没有结构化表单式 slot filling 引擎
- 没有通用的 approval workflow / authorization gate
- 没有把“等待用户授权”定义成一等运行态对象

所以它更像：

“一个支持自然对话追问、可持续多轮交互、可在长任务中吸收用户新输入的工具型 Agent Runtime”

而不是：

“一个内建审批工作流、授权关卡、表单补全状态机的流程编排系统”

---

## 2. 当前项目支持的 HITL 能力

### 2.1 Agent 可以主动向用户发起追问

主 Agent 默认注册了 `message` 工具，允许模型主动向当前用户发送消息。

这个工具的定位非常直接：

- 发送普通文本
- 发送带附件的消息
- 定向发到当前 channel / chat

也就是说，如果 Agent 发现：

- 缺参数
- 缺文件
- 缺用户决策
- 需要人工确认

它完全可以自己发一条消息给用户，例如：

- “还缺少材料参数，请补充。”
- “是否允许我执行这个 shell 命令？”
- “请上传几何文件后我再继续。”

关键实现：

- `nanobot/agent/tools/message.py`

这个工具并不会“阻塞等待用户”，它只是把消息发出去。后续是否继续，要靠用户的新消息再次进入系统。

---

### 2.2 用户回复后，可以沿着同一 session 继续

项目的消息模型基于 `InboundMessage` / `OutboundMessage` 与 `session_key`。

每条用户消息进入系统后，都会被绑定到一个会话：

- 默认 `session_key = "{channel}:{chat_id}"`
- 之后的消息会继续进入同一个会话历史

这意味着最基础的 HITL 是天然成立的：

1. Agent 问用户一个问题
2. 用户回复
3. 新回复会进入同一 session
4. Agent 可以在已有上下文上继续推理

关键实现：

- `nanobot/bus/events.py`
- `nanobot/agent/loop.py`

这其实就是“自然多轮追问式 HITL”。

---

### 2.3 支持 mid-turn injection：用户消息可中途注入当前运行中的任务

这是当前项目里最值得注意的一点，也是它相比很多普通 Agent Loop 更接近 HITL 的地方。

在 `AgentLoop` 里，系统专门维护了：

- `self._pending_queues`

注释已经写得很明确：

- 每个 session 有一个 pending queue
- 如果这个 session 当前已经有任务在运行
- 新来的用户消息不会立刻开一个新的竞争任务
- 而是优先路由进这个 pending queue

然后，在 `AgentRunner.run(...)` 的执行过程中，会在多个关键节点尝试 drain 这些 follow-up 消息，并把它们追加进当前 messages，继续后续迭代。

当前实现里会在这些时机尝试注入：

- 工具执行后
- 最终回复后但流式输出还没真正结束时
- 工具错误后
- LLM 错误后
- 空回复后
- 达到最大迭代后

这意味着：

如果 Agent 正在工作，用户突然补充了一条信息，例如：

- “刚才材料写错了，换成铝合金。”
- “我确认，可以继续执行。”
- “模型文件已经上传好了。”

系统可以把这条用户消息中途并入当前运行任务，而不是只能等整轮结束后重新开一轮。

这已经是一个非常实用的 HITL 能力。

关键实现：

- `nanobot/agent/loop.py`
- `nanobot/agent/runner.py`

相关测试也很多，说明这不是偶然行为，而是被明确设计和测试过的机制：

- `tests/agent/test_runner.py`

---

### 2.4 支持中断恢复，适合“问完用户后稍后继续”

HITL 的核心不是“能问”，而是“问完之后还能可靠继续”。

这个项目在这方面做了两层保护。

#### 保护 1：用户消息会尽早持久化

在 `_process_message()` 中，用户消息会在真正跑 Agent Loop 之前先写入 session。

目的很明确：

- 如果运行中途崩溃
- 至少用户输入不会丢

#### 保护 2：运行中 checkpoint 会落盘

AgentLoop 有两个关键元数据：

- `_RUNTIME_CHECKPOINT_KEY`
- `_PENDING_USER_TURN_KEY`

运行过程中会把：

- assistant 中间结果
- 已完成工具结果
- 未完成工具调用

写入 checkpoint。

下一次新消息到达时，系统会尝试：

- `_restore_runtime_checkpoint(session)`
- `_restore_pending_user_turn(session)`

把未完成的上下文恢复到 session 历史里。

这对 HITL 特别重要，因为很多“等待人工”场景并不是 5 秒内就会回复，而是可能：

- 过几分钟
- 过几小时
- 中途服务还可能重启

当前实现虽然没有显式的“awaiting approval”状态机，但它确实已经具备了“基于会话历史和 checkpoint 的恢复式继续”能力。

---

## 3. 当前项目不支持的 HITL 能力

这是最关键的边界分析。

### 3.1 没有显式的 `awaiting_user_input` / `awaiting_approval` 状态

我在 `nanobot/**/*.py` 里没有找到通用运行时层面的这些状态：

- `awaiting_user_input`
- `awaiting_approval`
- `awaiting_confirmation`
- `approval_state`
- `slot_filling_state`

这说明：

系统不会把“正在等用户确认”视为一个一等运行态对象。

当前等待用户，本质上只是：

- Agent 发出一条问题消息
- 这一轮结束
- 下次用户再回复
- 系统基于会话历史继续

或者：

- 消息中途注入当前运行任务

这是一种“自然会话式等待”，不是“显式状态机式等待”。

---

### 3.2 没有系统级审批拦截器

当前工具执行链路里，没有看到类似这样的通用机制：

- 在执行某类工具前自动弹出批准请求
- 未批准则禁止执行
- 批准后自动恢复被挂起的工具调用

主 Agent 是否会在危险操作前询问用户，主要取决于：

- 模型当前提示词
- 具体 skill 的写法
- 你给工具的设计约束

也就是说：

**项目允许你做“先问再做”，但系统不会强制所有危险工具都“必须先问”。**

如果你需要强审批，必须自己加一层。

---

### 3.3 没有结构化 Slot Filling 引擎

当前项目支持多轮追问，但这种追问仍然主要是：

- LLM 根据当前上下文自由决定该问什么

而不是：

- 系统根据 schema 明确维护“哪些字段还缺失”
- 逐项追问
- 做字段级验证
- 再恢复执行

仓库中的 CAE 相关文档里也明确把这件事列为“建议新增”的能力，而不是已有运行时能力。

这说明作者自己也知道：

- 现在可以靠提示词和对话机制做 PoC
- 但正式工程化版本需要显式 Slot Filling

---

### 3.4 子 Agent 不能直接做人机交互

`subagent` 在构造工具集时，代码里明确写了：

- no message tool
- no spawn tool

这意味着子 Agent：

- 不能直接问用户
- 不能直接等待用户回复

如果 subagent 执行到一半发现缺参数，它不能自己向用户发消息。

它只能：

1. 任务失败或返回结果给主 Agent
2. 主 Agent 收到后再问用户
3. 用户回复后，主 Agent 再决定是否重新派发子任务

所以 HITL 是主 Agent 层能力，不是 subagent 层能力。

---

## 4. 这套 HITL 是如何工作的

下面按运行机制拆开讲。

### 4.1 入口：用户消息进入 MessageBus

所有用户消息都会先进入：

- `MessageBus.inbound`

再由 `AgentLoop.run()` 消费。

这是系统统一的人机交互入口。

---

### 4.2 如果当前 session 没有任务在跑

消息会进入正常流程：

1. 读取 session
2. 组装历史
3. 运行主 Agent
4. 根据模型决策调用工具
5. 返回结果

如果模型发现需要用户输入，它有两种做法：

- 直接自然语言回答，问用户
- 显式调用 `message` 工具给用户发消息

两种方式都能形成 HITL。

---

### 4.3 如果当前 session 已经有任务在跑

此时就进入当前项目最有价值的机制：

- pending queue
- injection callback

新的用户消息会被放进当前 session 的 pending queue，而不是立刻启动另一个并发对话任务。

随后 `AgentRunner` 会在本轮执行中的关键节点尝试把这些 follow-up 消息取出来，追加进当前对话上下文，继续下一轮推理。

换句话说：

这不是简单的“等上一轮结束”，而是：

“让运行中的 Agent 有机会吸收人类的即时补充”

这正是一个轻量级 HITL runtime 的核心特征。

---

### 4.4 如果运行被中断

系统会依靠：

- session 历史
- runtime checkpoint
- pending user turn 标记

进行恢复。

这使得“问了用户但中途崩了”不会直接丢上下文。

---

## 5. 适合的 HITL 场景

当前实现适合以下场景。

### 5.1 信息补全型

例如：

- 缺少路径、文件、参数
- 缺少材料、边界条件、网格尺寸
- 缺少某个 API token 或仓库地址

Agent 可以问，用户回答，继续执行。

### 5.2 轻量确认型

例如：

- “是否继续执行这个命令？”
- “是否覆盖现有文件？”
- “是否切换到该分支？”

这种场景可以通过 prompt / skill 约束实现。

但注意，这种“确认”是软约束，不是系统硬门禁。

### 5.3 执行中补充型

例如 Agent 正在长时间搜索/分析时，用户又补充：

- “范围只看 `src/`”
- “不要改测试文件”
- “刚才那个路径错了，换成这个”

这类消息可以通过 mid-turn injection 被中途吸收。

### 5.4 长会话恢复型

例如：

1. Agent 问用户要材料参数
2. 用户半小时后才回复
3. 系统继续沿着同一 session 往下做

这类场景当前架构是支持的。

---

## 6. 不适合的 HITL 场景

下面这些场景，当前项目不能算“原生支持”。

### 6.1 强审批流

例如：

- 所有 shell 命令必须用户点击批准后才能执行
- 未批准之前任务处于正式 `awaiting_approval` 状态
- 批准后自动恢复被挂起的具体工具调用

当前没有这种统一机制。

### 6.2 结构化表单采集

例如：

- 系统明确知道还缺 `material`, `load`, `mesh_size`
- 每次只问一个字段
- 字段值类型校验
- 直到全部补齐再执行

这需要显式 slot filling 状态机，当前没有。

### 6.3 子 Agent 直接向人类追问

当前 subagent 没有 `message` 工具，所以不支持。

### 6.4 授权票据式操作恢复

例如：

- 任务暂停
- 发出批准卡片
- 用户点击批准
- 系统恢复同一个 suspended execution frame

当前没有这种挂起执行帧恢复能力。

---

## 7. 对“用户补充输入”和“用户授权”的分别判断

这是最容易混淆的地方。

### 7.1 用户补充输入：支持

支持方式包括：

- 主 Agent 直接追问
- 用户回复后继续同一 session
- 长任务中 follow-up 注入当前运行中
- 中断恢复

所以如果你的需求是：

- 缺参数就追问
- 用户补完再继续

当前项目是可以工作的。

### 7.2 用户授权：部分支持，但只是软支持

支持的部分是：

- Agent 可以主动向用户发“请确认”消息
- 用户回复“确认/继续/授权”后，Agent 可以继续后续动作

不支持的部分是：

- 没有系统强制的授权门
- 没有统一审批状态
- 没有自动恢复特定工具调用

所以如果你的需求是：

- “先问用户同意，再继续做下一步”

当前项目可以实现。

但如果你的需求是：

- “未获批准时系统层绝不允许某类操作被执行”

当前项目还不具备这种强保证。

---

## 8. 如果你要把 HITL 做成稳定工程能力，建议怎么扩展

如果你只是 PoC，当前能力已经够用。

如果你要做正式工程化流程，建议新增下面这些层。

### 8.1 明确的会话状态机

建议引入类似：

- `idle`
- `awaiting_user_input`
- `awaiting_approval`
- `running`
- `failed`
- `completed`

这样系统就能显式知道当前为什么停住。

### 8.2 结构化 approval gate

建议在工具执行前增加统一拦截层：

- 某些工具标记为 `requires_approval=True`
- 命中后不直接执行
- 生成 approval request
- 等用户同意后再恢复执行

### 8.3 结构化 slot filling

建议增加：

- required fields 列表
- 已收集字段
- 缺失字段
- 字段级校验器

这样追问就不是完全靠 LLM 自由发挥。

### 8.4 用户事件到运行事件的映射层

例如：

- 用户回复“继续”
- 用户点击“批准”
- 用户上传文件

这些都可以先被映射成结构化事件，再驱动运行状态机。

---

## 9. 对 CAE / 工程 Agent 的直接启示

如果你要基于这个项目做 CAE、EDA、仿真、脚本执行类 Agent，我建议这样理解当前 HITL：

### 可以直接利用的部分

- 参数不足时追问
- 用户补充后继续
- 长任务中补充消息注入
- 会话级恢复

### 不能直接依赖的部分

- 脚本执行审批
- 危险命令授权
- 结构化参数采集
- 批准后原地恢复具体执行节点

所以短期方案可以是：

- 先用 prompt + tool schema + 多轮对话把 HITL 跑通

中期如果要正式上线，建议新增：

- Slot Filling Manager
- Approval Manager
- Execution State Machine

## 10. 一个最贴近当前实现的 HITL 示例

下面是当前项目里最自然的一种工作方式。

### 场景：执行脚本前需要用户确认

1. 用户说：“帮我批量修改这些 CAD 文件并导出网格。”
2. Agent 分析后发现：
   - 缺少网格尺寸
   - 后续会执行 shell 命令
3. Agent 先问：
   - “请补充网格尺寸。”
4. 用户回复：
   - “2mm”
5. Agent 继续分析，准备执行命令前再问：
   - “接下来会调用网格生成脚本，是否继续？”
6. 用户回复：
   - “继续”
7. Agent 执行后续工具调用

这个流程当前是可实现的。

但这里的“是否继续”只是会话层面的逻辑约束，不是运行时硬拦截。

---

## 11. 一句话总结

当前 `nanobot` 的 HITL 能力本质上是：

**基于 session、多轮对话、message 工具、mid-turn message injection 和 checkpoint 恢复机制，实现的人机协作式执行。**

它已经足够支持：

- 追问
- 补充输入
- 轻确认
- 执行中补充
- 断点恢复

但还不等于：

- 显式审批流
- 结构化 slot filling
- 系统级授权门
- 强状态机恢复

如果你的目标是“先做 PoC”，当前机制已经够用。

如果你的目标是“做工业级人机协同流程”，需要在当前 runtime 之上再加一层明确的 HITL orchestration。
